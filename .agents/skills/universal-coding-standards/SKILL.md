# Universal Coding Standards

Rules are classified as **Mandatory** (must follow; violations should block merge)
or **Recommended** (strongly encouraged; exceptions need justification).

---

## 1. Communication & Runtime Text

### 1.1 Runtime Text MUST Be Plain English (Mandatory)

All **runtime text** — exception messages, error strings, log messages, API
responses, user-facing output generated by the program — MUST be written in
plain English. Non-English text can break console encoding and is inaccessible
to international tooling and collaborators.

```python
# Correct
raise ValueError("Configuration file path cannot be empty")

// Correct
throw new InvalidOperationException("Database connection is not open");

# Incorrect
raise ValueError("配置文件路径不能为空")
```

Emoji, non-printable characters, and other non-plain-text sequences are forbidden
in runtime text. They cause encoding issues in logs, terminals, and monitoring
systems.

### 1.2 Comments May Use Any Language (Recommended)

Code comments may be written in whatever language the team is most comfortable
with. Prefer the project's dominant comment language, but clarity for the team
always takes priority over language choice.

## 2. Dependency Management

### 2.1 Never Modify Dependencies Without Authorization (Mandatory)

Project dependencies — packages, libraries, frameworks, plugins — MUST NOT be
added, removed, or version-bumped without explicit user authorization. Before
using a new library, ask the user to evaluate its supply-chain security and add
it themselves. Unauthorized dependency changes introduce security, licensing,
and compatibility risk that the agent cannot fully assess.

This does NOT mean dependencies are forbidden. It means the **user** decides
what to trust. The agent's role is to identify the need and recommend, not to
install.

```
// Agent workflow when a dependency would help:
1. Identify that a library would solve the problem better than hand-writing
2. Tell the user: name, purpose, why it fits, and its trustworthiness signals
3. Let the user evaluate, install, and confirm it is available
4. Only then write code that uses it
```

### 2.2 Verify Before Assuming a Library Exists (Mandatory)

NEVER assume a library or framework is available. Before writing code that
depends on one, inspect the project's dependency manifest (e.g., `package.json`,
`Cargo.toml`, `go.mod`, `.csproj`, `requirements.txt`, `Gemfile`) to confirm it
is already referenced. Writing code against an unavailable dependency wastes
time and can break the build.

### 2.3 Prefer Trusted Libraries Over Hand-Written Solutions (Recommended)

If a well-known, actively maintained, widely adopted library exists for a given
task, do not hand-write the equivalent functionality. Instead, recommend it to
the user (per rule 2.1) and let them decide. Hand-rolled implementations of
common tasks (encryption, parsing, date/time handling, HTTP clients) are a
leading source of bugs and vulnerabilities.

## 3. Safety & Security

### 3.1 Destructive Commands Are Strictly Forbidden (Mandatory)

The following categories of commands MUST NEVER be executed:

- Destructive filesystem operations: `rm -rf`, `dd`, `mkfs`, destructive
  redirections that overwrite critical files
- Destructive version control: `git push --force`, `git reset --hard`,
  `git clean -fd`, branch deletion without confirmation
- System-level destructive actions: `shutdown`, `reboot`, `kill -9` on
  unknown processes, `chmod 777` on sensitive paths

If a destructive operation is genuinely needed, explain why and ask the user to
perform it manually. Never bypass safety checks (`--no-verify`, `--no-gpg-sign`,
`--force`) unless the user explicitly requests it.

### 3.2 Secrets Must Never Be Exposed (Mandatory)

Secrets, credentials, tokens, API keys, connection strings, private keys, and
any other sensitive information MUST NEVER be:

- Hard-coded in source files
- Written to log output or stdout/stderr
- Committed to version control
- Included in comments or documentation
- Passed as command-line arguments (visible in process lists)

```python
# Correct: read from environment
api_key = os.environ["API_KEY"]

# Correct: read from a secret manager
api_key = secret_manager.get("stripe/api_key")

# Incorrect: hard-coded
api_key = "sk_live_4eC39HqLyjWDarjtT1zdp7dc"
```

When generating example code, use placeholder values like `"your-api-key-here"`
or environment variable references. Never use real credentials, even as examples.

### 3.3 Ask When Uncertain (Mandatory)

When business logic, architectural decisions, requirements, or intended behavior
are unclear, STOP and ask the user. Never guess. Incorrect assumptions about
business rules cause bugs that are disproportionately expensive to fix later.

This applies especially to:
- Domain-specific validation rules
- Edge-case handling (what should happen when X is empty?)
- Architectural patterns (which layer owns this logic?)
- Data modeling (is this field required? what's the max length?)

## 4. Code Quality & Project Conventions

### 4.1 Study Existing Code Before Writing (Mandatory)

Before writing any code, examine existing code of the **same type** in the
project — same layer, same pattern, same category. Understand the established:

- Structure and file organization
- Naming conventions
- Design patterns and architectural choices
- Framework/library usage patterns
- Error handling approach

Never write code in isolation. New code that ignores existing conventions creates
a fractured codebase where each file follows different rules.

### 4.2 Follow Established Project Conventions (Mandatory)

When modifying existing code, strictly follow the conventions already in use.
Do not introduce new patterns, naming styles, or architectural approaches unless
explicitly requested. Consistency within a codebase matters more than any
external style guide. If the project uses `snake_case`, you use `snake_case`.

### 4.3 Preserve Existing Comments (Mandatory)

Existing comments MUST NOT be deleted when modifying code, unless removing them
is explicitly authorized. Comments carry institutional knowledge — why a
workaround exists, what edge case a branch handles, what constraint a magic
value satisfies — that the current diff may not make obvious.

When adding new code alongside existing comments, keep them intact and add your
own only where necessary.

### 4.4 Understand Requirements Before Coding (Mandatory)

Before writing any code, ensure that the business context and architecture are
thoroughly understood. Read the surrounding code, review related issues or
documentation, and confirm the approach with the user when appropriate. Code
written without understanding the "why" is code that will need to be rewritten.

### 4.5 Search for Documentation When Unfamiliar (Recommended)

When encountering unclear or unfamiliar APIs, patterns, or library functions, use
web search to find official documentation and examples (if web search is
available). Relying on reflection, decompilation, or trial-and-error to reverse-
engineer library behavior is slow and error-prone. Official docs and community
examples provide context that raw source code alone does not.

### 4.6 Comments Explain Why, Not What (Recommended)

Code should be self-documenting about **what** it does through clear naming and
structure. Comments should explain the **why** — motivation, constraints, edge
cases, non-obvious trade-offs.

```python
# Correct: explains the motivation
# Skip validation for system accounts so they can use special characters
# that normal username rules would reject.
if not account.is_system_account:
    validate_username(account.username)

# Incorrect: restates what the code already says
# Check if account is not a system account
if not account.is_system_account:
    # Validate the username
    validate_username(account.username)
```

### 4.7 Avoid Commented-Out Code (Recommended)

Do not leave commented-out code in production. During active development it is
tolerable, but it should be removed before merging. Version control preserves
the old code; inline comments do not.

### 4.8 Avoid Magic Numbers (Recommended)

Replace hardcoded numeric literals with named constants. Exceptions: `0`, `1`,
`-1` in trivial contexts (loop counters, array indices, initial values) are
exempt.

```python
# Correct
MAX_RETRY_COUNT = 3
for attempt in range(MAX_RETRY_COUNT):
    ...

# Incorrect
for attempt in range(3):  # what does 3 mean?
    ...

# Allowed: 0, 1, -1 in obvious contexts
first_item = items[0]
```

### 4.9 Avoid Double Negatives (Recommended)

Write conditions in the positive form when possible. Double negatives force the
reader to mentally invert the logic, increasing the chance of errors.

```javascript
// Preferred: positive form
if (isActive) {
    process();
}

// Preferred: straightforward negative
if (!isValid) {
    return;
}

// Avoid: double negative
if (!isInactive) {
    process();  // wait, so we process when it's... active?
}
```

## 5. Process & Planning

### 5.1 No Unsolicited Plans in the Project (Mandatory)

Unless the user explicitly requests it, do NOT write planning documents, design
docs, implementation roadmaps, or any similar planning artifacts into the
project directory. Planning happens in the conversation or in designated
planning tools. Writing plans into the project tree clutters the repository and
creates stale documentation that misleads future readers.

### 5.2 Prefer Clarity Over Cleverness (Recommended)

Write code that is obvious to the next reader. Avoid unnecessarily compact or
tricky constructs that prioritize brevity over understanding. The next reader
may be you, six months from now, debugging a production issue at 3am.

```python
# Preferred: clear and explicit
active_customers = [c for c in customers if c.is_active]

# Avoid: too clever
active = [c for c in cs if c.a]
```

## Quick Review Checklist

Check in order:

1. Runtime text (errors, logs, messages) in plain English, no emoji?
2. Dependencies unchanged? New library usage authorized by user?
3. Library availability verified in project manifest before use?
4. No destructive commands executed?
5. Secrets absent from source code, logs, and comments?
6. Uncertain requirements clarified with user (not guessed)?
7. Existing code of same type studied before writing?
8. Project conventions followed (naming, structure, patterns)?
9. Existing comments preserved?
10. Business context and architecture understood?
11. Unfamiliar APIs documented (web search used if available)?
12. Comments explain why, not what?
13. No commented-out code remaining?
14. No magic numbers (except 0/1/-1 in trivial contexts)?
15. Conditions in positive form, no double negatives?
16. No unsolicited planning documents written to project?
17. Code clear and direct, not overly clever?
